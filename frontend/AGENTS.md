## Frontend AGENTS — Blog4uSlf

Этот файл описывает правила и лучшие практики для **frontend-части** проекта (`frontend/`), построенной на **Nx + Angular 21 + NgRx + Taiga UI**, с использованием подхода **FSD**.

---

## 1. Архитектура фронтенда

- **Nx-монорепозиторий**
  - Основное приложение: `apps/blog-4u-slf`.
  - Фичи/страницы/виджеты/entites выносим в библиотеки под `libs/` (например, `libs/pages/home`, `libs/pages/posts`).
  - Для новых частей UI по возможности создаём отдельную библиотеку/страницу, а не кладём всё в `apps/blog-4u-slf`.

- **FSD (Feature-Sliced Design)**
  - Стремимся разделять код по слоям/фичам, а не по типу артефактов (`components`, `services` и т.д.).
  - Страницы (`pages`) агрегируют фичи и UI-компоненты, не содержат низкоуровневой бизнес-логики.
  - Переиспользуемые компоненты и логика по мере роста проекта могут выделяться в отдельные слои/библиотеки.

- **Структура приложения**
  - Входная точка: `apps/blog-4u-slf/src/main.ts`, корневой компонент/конфиг — в `app/`.
  - Маршруты приложения описываем в `app.routes.ts` / `app.routes.server.ts`, лениво подгружая страницы/фичи.

---

## 2. Angular: компоненты, модули и маршрутизация

- **Компоненты**
  - Используем современные подходы Angular 21 (standalone-компоненты, Signals, где это оправдано).
  - Компонент отвечает за отображение и простую UI-логику; бизнес-правила выносим в сервисы/NgRx.
  - Стили для компонента храним рядом (`*.scss`), избегаем глобальных стилей, если это не layout/темизация.
  - В названии компонента не используем устаревший префикс `Component` (PageComponent к примеру должен быть Page).
  - Для инъекции зависимостей стараемся всегда использовать функцию `inject`, вместо конструктора

- **Сервисы**
  - В названии сервиса не исользуем устаревший префикс `Service`.
  - Для инъекции зависимостей стараемся всегда использовать функцию `inject`, вместо конструктора.

- **Директивы**
  - Для инъекции зависимостей стараемся всегда использовать функцию `inject`, вместо конструктора.
  - В названии директивы не используем устаревший префикс `Directive`.

- **Пайпы**
  - Для инъекции зависимостей стараемся всегда использовать функцию `inject`, вместо конструктора.
  - В названии конвеера(Пайпа) не используем устаревший префикс `Pipe`.

- **Маршруты**
  - Каждая страница (например, `home-page`, `post-page`) подключается через маршруты в приложении.
  - Используем lazy loading для страниц и фичей, чтобы не раздувать initial bundle.

- **Шаблоны (`*.html`)**
  - Логику в шаблоне держим простой: без тяжёлых вычислений и «магии» в привязках.
  - При повторяющихся кусках UI выносим их в отдельные компоненты.
  - Используем новый синтаксис control-flow, вместо устаревших директив *ngIf, *ngFor и других директив, элементов.

## 3. Состояние: NgRx и Signals

- **NgRx (`@ngrx/effects`, `@ngrx/operators`, `@ngrx/signals`)**
  - Применяем NgRx там, где есть **общий/долгоживущий стейт**, асинхронщина, side effects (загрузка постов, фильтры и т.п.).
  - Локальное состояние компонента (форма, флаги показа) держим в самом компоненте или через Signals.
  - Эффекты должны быть **чистыми**: только обработка action → side effect → новые action.

- **Signals**
  - Используем Signals для локального или среднеуровневого состояния, когда NgRx кажется избыточным.
  - Следим, чтобы сигналами не дублировать глобальный стейт, уже управляемый NgRx.

## 4. UI и UX: Taiga UI и стили

- **Taiga UI**
  - Основной UI-kit: `@taiga-ui/cdk`, `@taiga-ui/core`, `@taiga-ui/kit`, `@taiga-ui/icons`, `taiga-ui`.
  - Предпочитаем использовать компоненты Taiga UI вместо написания сложных контролов с нуля.
  - Настройки темы, тёмный/светлый режим и общие layout-паттерны по возможности централизуем.

- **Стилизация**
  - Глобальные стили: `src/styles.scss` — только базовые вещи (reset, шрифты, root-переменные).
  - Локальные стили компонентов (`*.scss`) — без утечек в глобальную область имён (избегаем глобальных селекторов).
  - Поддерживаем **изолированные и читаемые** стили, без жёсткой завязки на структуру DOM других компонентов.

## 5. Работа с API

- **HTTP-клиент**
  - Для запросов к backend используем новый ResourceApi Angular.
  - Выносим работу с API в отдельные сервисы/слой/стейты (не вызываем `HttpClient` напрямую из компонентов).

- **Модели и типы**
  - Для DTO/ответов от API определяем **типизированные интерфейсы**/типы в отдельных файлах.
  - Следим за синхронизацией контрактов с backend (при изменении API обновляем типы и использование).

- **Обработка ошибок**
  - Ошибки запросов обрабатываем централизованно (interceptors, эффекты), а не в каждом компоненте.
  - Для пользовательских сообщений используем унифицированный подход (например, сервис уведомлений поверх Taiga UI).

## 6. Тестирование и качество кода

- **Unit-тесты (Jest)**
  - Конфиг общих тестов: `jest.config.ts`, `jest.preset.js`.
  - Для приложений и библиотек используем их локальные `jest.config.*` + `test-setup.ts`.
  - Покрываем:
    - Ключевые компоненты и их логику.
    - Сервисы, хранящие бизнес-правила.

- **E2E-тесты (Cypress)**
  - E2E-проект: `apps/blog-4u-slf-e2e`.
  - Тесты в `src/e2e`, page-objects в `support/app.po.ts`.
  - Пишем сценарии, покрывающие критичные пользовательские флоу (просмотр списка постов, просмотр конкретного поста и т.д.).

- **Линтинг и форматирование**
  - Линтер: ESLint (`eslint.config.mjs`, `angular-eslint`).
  - Форматер: Prettier.
  - Перед коммитом желательно прогонять линт и тесты для изменённых проектов (через `nx lint`, `nx test`).

## 7. Работа с Nx

- **Запуск задач**
  - Dev-сервер: `npx nx serve blog-4u-slf`.
  - Сборка: `npx nx build blog-4u-slf`.
  - Тесты: `npx nx test <project>` (например, `home-page`, `post-page`, `blog-4u-slf`).

- **Новые проекты**
  - Используем генераторы Nx (`nx g`) для создания новых приложений/библиотек.
  - Стараемся не создавать «ручных» структур в обход Nx, чтобы не ломать зависимости и graph.

- **Производительность**
  - По возможности используем кеширование и параллельный запуск задач Nx.
  - Следим за размером бандла, при необходимости выносим тяжелые/редко используемые части в отдельные lazy-модули.

## 8. Общие рекомендации для frontend-изменений

- При добавлении новой функциональности:
  - Сначала определяем, где ей место в FSD-структуре (страница, фича, общий UI и т.д.).
  - Продумываем, нужен ли глобальный стейт (NgRx) или достаточно локального/Signals.
  - Проверяем, не противоречит ли решение текущему UX-подходу и использованию Taiga UI.

- При изменении API/контрактов:
  - Обновляем типы и сервисы работы с API.
  - Проверяем, какие страницы/фичи завязаны на этот контракт через Nx graph (при необходимости).

Дальнейшая детализация code-style, соглашений по именованию и структуре модулей/директорий может быть добавлена поверх этого файла.
